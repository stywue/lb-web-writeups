# Web Challenge 3: TalentFlow

## Что нужно было сделать
Получить флаг из внутренней инфраструктуры сервиса, используя цепочку уязвимостей вокруг загрузки DOCX и FTP.

## Коротко про идею атаки
Цепочка была такая:
1. Загружаем «нормальный» DOCX, чтобы понять формат.
2. Подменяем часть содержимого на полезную нагрузку.
3. Используем FTP bounce/command injection, чтобы достучаться до внутреннего сервиса.
4. Ждём, пока воркер обработает задачу.
5. Забираем результат с флагом.

## Шаг 1. Разведка
Посмотрел поведение приложения и формат входного файла (DOCX).
Цель — не ломать сразу всё, а понять, где именно данные из документа попадают в backend-процессы.

## Шаг 2. Подготовил валидный DOCX
Сначала загрузил обычный файл без вредоносного содержимого, чтобы убедиться, что пайплайн обработки работает и есть стабильная точка входа.

## Шаг 3. Вставил payload
Собрал полезную нагрузку в формате, который уходит дальше по цепочке (через FTP-команды/пересылку), чтобы заставить сервис обратиться к внутреннему ресурсу.

## Шаг 4. Выполнил FTP bounce
Дальше отправил запрос так, чтобы через FTP-механику проброситься к нужному внутреннему хосту/порту и инициировать выполнение нужной команды/запроса.

## Шаг 5. Подождал обработчик
После отправки нужно было немного подождать воркер: он не сразу возвращает результат.

## Шаг 6. Скачал результат
После завершения обработки получил итоговый артефакт/ответ, где и лежал флаг.

## Флаг
```text
LB{78844f003b0dbe51b67791cc295bc815}
```

## Почему это сработало
- Недостаточная валидация данных из загружаемого файла.
- Возможность злоупотребить FTP-логикой для доступа к внутренним сервисам.
- Слабая сегментация между внешним контуром и внутренними компонентами.

## Как исправить
1. Жёстко валидировать содержимое документов и запрещать опасные конструкции.
2. Отключить/ограничить FTP-команды, которые позволяют bounce-сценарии.
3. Изолировать воркеры в отдельной сети без доступа к чувствительным внутренним сервисам.
4. Добавить allowlist адресов и портов для исходящих соединений.

## Итог
Задание решалось не одной дырой, а аккуратной цепочкой. Главный принцип: шаг за шагом проверить гипотезу, зафиксировать рабочий путь и только потом автоматизировать.
