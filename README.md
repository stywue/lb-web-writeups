# Web Challenge 1: Chess — простой пошаговый райтап

## Что было в задании
Нужно «обыграть Stockfish» через WebSocket-сервер с шахматами.

Сначала кажется, что это нереально, но подвох был не в шахматной силе, а в логике сервера.

## Шаг 1. Проверил, как работает обычный клиент
Запустил обычный клиент и посмотрел порядок сообщений.

```bash
python3 play_challenge.py
```

Нормальный сценарий такой:
1. Ждём старт игры.
2. Отправляем один ход.
3. Ждём ответ сервера.
4. Дальше следующий ход.

## Шаг 2. Нашёл подозрительный режим в `solve.py`
В `solve.py` есть режим `--burst`: он отправляет **сразу несколько ходов подряд**, не дожидаясь ответа между ними.

Идея: если сервер плохо проверяет очередь ходов, можно сделать несколько белых ходов подряд и не дать чёрным походить.

## Шаг 3. Выбрал короткий мат
Взял классический мат в 4 хода (Scholar’s Mate):
1. `e2e4`
2. `d1h5`
3. `f1c4`
4. `h5f7`

## Шаг 4. Запустил эксплойт
```bash
python3 solve.py --burst e2e4 d1h5 f1c4 h5f7
```

Скрипт быстро отправляет все 4 хода подряд.

## Шаг 5. Почему это сработало
Сервер принимал ходы без жёсткой проверки «чья сейчас очередь». В итоге мои белые ходы обрабатывались подряд, а движок не успевал сделать чёрный ход между ними.

## Что именно уязвимо
- Нет строгой серверной проверки очереди хода.
- Асинхронная обработка сообщений позволяет «залить» пачку ходов.
- Нет ограничения/троттлинга на частоту отправки.

## Как чинить
1. Хранить текущий ход (white/black) на сервере.
2. Отклонять любой ход «не своей очереди».
3. Обрабатывать следующий ход только после полного завершения предыдущего.
4. Добавить rate limit на клиента.

## Итог
Задание было не про шахматы, а про race condition/десинхронизацию по WebSocket. Достаточно было быстро отправить выигрышную последовательность ходов в `--burst`.
